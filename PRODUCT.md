# Product

## Register

product

## Users

Symphony is used by engineers and technical operators who manage autonomous
implementation runs across trusted local repositories. They need to see which
tasks are queued, running, blocked, in review, or done without leaving the
dashboard or switching back to terminal tracker commands.

## Product Purpose

Symphony turns tracker work into isolated agent workspaces and unattended
implementation sessions. The dashboard exists to make orchestration state,
queue health, run progress, and handoff evidence visible enough that an
operator can supervise the system at a higher level.

## Brand Personality

Calm, technical, exact. The interface should feel like a dependable operator
console: dense enough for real work, restrained enough for long sessions, and
clear about state changes that matter.

## Anti-references

Avoid marketing-style SaaS dashboards, oversized hero metrics, decorative dark
mode, generic gradient panels, and playful task-board styling. The product
should not hide operational detail behind chrome or make routine actions feel
ceremonial.

## Design Principles

- State before decoration: visual emphasis should clarify workflow status,
  claim eligibility, and review/blocking conditions.
- Keep operator flow in place: selecting, inspecting, and queueing work should
  happen inline without route changes or terminal context switches.
- Earn density: compact layouts are acceptable when hierarchy, spacing, and
  focus states keep the board scannable.
- Use familiar controls: standard board columns, toggles, chips, and pane
  patterns should carry the interaction model.
- Make automation visible: queued and claimed work should have obvious,
  auditable transitions without exaggerating motion.

## Accessibility & Inclusion

Target WCAG 2.1 AA contrast for text and controls. All interactive cards and
queue targets need keyboard focus states, pointer hover states, and visible
selected/drag feedback that does not rely on color alone.
